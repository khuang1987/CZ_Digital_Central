# 测量阶段发现的问题
- 数据与口径
  - 数据来源分散、缺少统一“单一真实来源”
    - Power BI、Excel、SFC、OEE、MES、邮件、SharePoint 各自维护数据
    - 字段与口径不统一，报表结果不一致
  - 指标口径与业务规则不透明
    - KPI/计算公式分散在脚本、个人 Excel、口头约定中
    - 同一指标在不同报表中数值不同，质疑与解释频发
  - 主数据与维度表不完备且不一致
    - 产品、工序、组织、客户等主数据由不同系统维护
    - 存在缺失、重复、历史版本混用，聚合分析口径不一致
  - 时间相关口径不统一（日历/班次/时区/节假日）
    - 各系统对班次、跨日、节假日的处理方式不同
    - 时间戳粒度和含义不一致，交期/产能/稼动率等指标存在系统性偏差

- 过程与质量
  - 缺乏数据质量规则与监控（缺失/重复/异常值）
    - 没有统一的数据质量标准和自动校验机制
    - 依赖人工“看数”发现问题，异常暴露滞后
  - 无系统化一致性校验与对账
    - 缺少跨系统、跨层级的自动对账与告警
    - 异常进入报表后难以及时发现和追溯
  - 手工录入与线下表格占比高，流程不闭环
    - 关键数据通过邮件、Excel、手工 A3 报告维护
    - 审批/修改记录不完备，责任与历史难追溯

- 报表与使用
  - 报表与指标体系碎片化、重复建设
    - 各部门按本地需求各自搭建报表
    - 同一主题存在多份相似但不一致的仪表板
  - 权限与数据安全治理不足
    - 缺少统一的权限模型和审批流程
    - 既存在越权访问风险，也存在因过度收紧导致取数绕过系统的行为

- 刷新与运维
  - 报表刷新机制依赖全量与人工触发
    - 以全量刷新为主，缺少增量、重跑与补数机制
    - 数据量增长后刷新时间过长或失败，形成“报表不可用窗口”
  - 配置分散、缺少变更审计与回滚机制
    - 报表配置与查询逻辑分散在脚本、多份 pbix、参数文件中
    - 线上数值变化难以定位是哪次修改导致，排查和回滚困难
